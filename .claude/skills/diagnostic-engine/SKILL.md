# 진단 문항 & 채점 엔진 설계

축을 측정 가능한 문항으로 바꾸고, 응답을 유형 코드로 환산한다. 산출물은 웹 테스트가 그대로 소비하는 **데이터 계약**이므로 스키마를 엄격히 지킨다.

## 1. 데이터 스키마

문항·유형·채점 데이터는 `references/data-schema.md`가 정의한 JSON 구조를 정확히 따른다. **반드시 이 파일을 먼저 읽어라.** 필드명·중첩 구조를 임의로 바꾸면 웹 빌더가 깨진다.

## 2. 문항 작성 원칙

- **축당 3~5문항**: 한 축을 여러 각도로 측정해 단일 문항 편향을 줄인다. 트랙당 12~20문항이 적정(4축 × 3~5).
- **구체적 상황 제시**: "리스크를 어떻게 보나요?"(추상) ❌ → "10% 싸지만 권리관계가 복잡한 물건과, 시세대로지만 깔끔한 물건 중 무엇을 택하나요?"(구체) ✅.
- **유도 금지**: 한 선택지가 명백히 우월해 보이면 안 된다. 모든 선택지가 합리적 투자자의 선택이어야 한다.
- **선택지 = 극 매핑**: 각 선택지는 해당 축의 한 극 코드에 점수를 준다. 2지선다(±2) 또는 4점 리커트(강함2/약함1) 방식 중 택일하고 일관되게 적용한다.

## 3. 경매 트랙 필수 변수

경매 문항에는 사용자 지정 5개 변수를 직접 묻는 문항을 포함한다:
- **투자자금**: 가용 자금 구간 (예: ~5천 / 5천~1.5억 / 1.5억~3억 / 3억+)
- **투자용도**: 실거주 / 임대수익 / 단기매각
- **투자지역**: 연고지 위주 / 수도권 / 전국
- **주택보유상황**: 무주택 / 1주택 / 다주택 (세금·대출규제 영향)
- **경매경험**: 입찰·낙찰 경험 횟수 (초보 0~1 / 중급 2~5 / 숙련 6+)

이들은 (a) 축 판별 점수에 기여하고 동시에 (b) `profile` 객체로 저장되어 활용 가이드 개인화에 쓰인다. 두 역할을 스키마에 분리 표기한다.

## 3-1. 규제 변수 & 규제 경고 플래그

부동산 규제(대출제한·세금중과·규제지역)는 보유주택수·지역에 따라 의사결정을 크게 바꾸므로 경고로 반영한다. **규제 수치는 절대 기억으로 쓰지 말고 `_workspace/00_regulations.md`(출처·신뢰도 표기된 grounding 문서)만 근거로 한다.** 그 문서가 없으면 규제 플래그를 만들지 말고 리더에게 알린다.

- **규제 profile 입력 추가**: `ownership`(무주택/1주택/다주택)에 더해 `region_regulated`(대상 지역이 규제지역인지: 규제/비규제/모름)를 매매·경매 양 트랙 profile_inputs에 추가한다. 이들은 축 점수에 기여하지 않고(0점) flags 평가에만 쓴다 → 유형 체계 불변.
- **규제 플래그**: `00_regulations.md`의 F절(플래그 매핑 제안)을 기반으로 flags를 만든다. 예: 다주택+규제지역+추가구입 → 주담대 금지(LTV 0%) 경고, 다주택+조정지역 → 취득세 8~12%·양도세 중과 경고.
- **신뢰도 전파**: grounding 문서가 `[불확실]`로 표기한 항목(예: 양도세 유예 종료, 6·30 신규지정 범위)은 플래그 문구에 "확인 필요"를 포함한다. 단정하지 않는다.
- **경매 특수 플래그(중요)**: 경매+규제지역은 두 축으로 분리한다 — (1) 대출규제는 경매도 동일 적용(자금계획 경고), (2) 토허 실거주의무·갭투자차단·자금조달계획서는 경매 면제(규제회피 가능). `00_regulations.md` D절 근거.

## 4. 채점 알고리즘

1. 각 축마다 양극 점수를 합산한다.
2. 우세 극의 코드를 채택한다. **동점(tie)이면** 사전에 정한 기본 극(축 정의의 첫 번째 극)을 채택하고, 결과에 "경계형(borderline)"임을 표시한다.
3. 축 순서대로 극 코드를 이어 붙여 유형 코드를 만든다(예: `M-SHLE`). 접두(M-/A-)는 트랙에서.
4. 경매는 profile 변수(보유상황·경험)에 따라 경고/안내 플래그를 추가로 산출한다(예: 다주택+단기매각 → 양도세 중과 플래그).

채점 규칙은 사람이 읽는 `_workspace/02_scoring.md`로도 문서화한다 — 웹 빌더가 이걸 보고 동일하게 JS를 구현하고, QA가 이걸로 재현 검증한다. 모호함 없이 결정적으로 쓴다.

- `_workspace/02_gyeongmae.json` — 경매 트랙 (스키마 준수, 경매 변수 포함)

샘플 골든 케이스를 반드시 포함하라 — QA와 웹 빌더가 채점 일치를 검증하는 기준점이 된다.
